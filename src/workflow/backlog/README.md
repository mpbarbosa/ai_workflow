# Workflow Execution Backlog

This directory contains issue reports from workflow automation script executions.

## Directory Structure

```
backlog/
├── README.md (this file)
├── workflow_YYYYMMDD_HHMMSS/     # Individual workflow run
│   ├── WORKFLOW_SUMMARY.md        # Overview of entire run
│   ├── step0_Pre_Analysis.md      # Step 0 issues/findings
│   ├── step1_Update_Documentation.md
│   ├── step2_Consistency_Analysis.md
│   ├── step3_Script_Reference_Validation.md
│   ├── step4_Directory_Structure_Validation.md
│   ├── step5_Test_Review.md
│   ├── step6_Test_Generation.md
│   ├── step7_Test_Execution.md
│   ├── step8_Dependency_Validation.md
│   ├── step9_Code_Quality_Validation.md
│   ├── step10_Context_Analysis.md
│   └── step11_Git_Finalization.md
└── archive/                        # Resolved workflow runs (optional)
```

## How to Use

### 1. Review Latest Workflow Run

```bash
# Find the most recent workflow run
ls -lt backlog/ | head -5

# View the summary
cat backlog/workflow_YYYYMMDD_HHMMSS/WORKFLOW_SUMMARY.md
```

### 2. Address Issues by Priority

Each step report contains:
- **Issue count** and severity
- **Detailed findings** with file references
- **Raw output** from validation tools

Work through issues systematically:
1. Critical issues first (blocking problems)
2. High priority (important improvements)
3. Medium priority (quality enhancements)
4. Low priority (nice-to-have optimizations)

### 3. Track Progress

As you resolve issues:
- Check off completed items
- Update relevant documentation
- Re-run affected validation steps
- Consider archiving resolved runs

### 4. Archive Completed Runs

Once all issues from a workflow run are resolved:

```bash
mkdir -p backlog/archive
mv backlog/workflow_YYYYMMDD_HHMMSS backlog/archive/
```

## Report Contents

### WORKFLOW_SUMMARY.md
- Overall execution status
- All steps completion status
- Links to individual step reports
- Change analysis summary

### Individual Step Reports
Each step report includes:
- Timestamp and workflow run ID
- Issues found count
- Detailed findings with context
- Raw validation output
- File/line references where applicable

## Best Practices

### Regular Maintenance
- Review new workflow runs after each execution
- Address critical issues before next development cycle
- Archive resolved runs monthly
- Clean up very old archives (6+ months)

### Integration with Development
- Run workflow before major commits
- Review backlog during sprint planning
- Use reports for code review preparation
- Track recurring issues for systematic fixes

### Issue Categorization
When reviewing reports, categorize issues as:
- **Bug:** Something is broken or incorrect
- **Enhancement:** Improvement opportunity
- **Documentation:** Missing or outdated docs
- **Test Coverage:** Missing or inadequate tests
- **Quality:** Code quality improvements

## Workflow Automation Script

These reports are generated by:
```bash
./src/workflow/execute_tests_docs_workflow.sh
```

**Version:** 2.0.0 (latest - fully modularized with 25 modules)

**Previous Versions**:
- v1.3.0: Initial backlog tracking
- v1.4.0: Added summaries directory
- v1.5.0: Added logs directory with AI session tracking
- v2.0.0: Complete modularization (25 modules: 12 libraries + 13 steps)

### Script Modes
- **Interactive** (default): Prompts for confirmations, allows AI analysis
- **Automatic** (`--auto`): No prompts, skips optional AI steps
- **Dry-run** (`--dry-run`): Preview actions without execution

### Example Usage
```bash
# Standard interactive run
./src/workflow/execute_tests_docs_workflow.sh

# Automatic mode for CI/CD
./src/workflow/execute_tests_docs_workflow.sh --auto

# Preview without changes
./src/workflow/execute_tests_docs_workflow.sh --dry-run
```
